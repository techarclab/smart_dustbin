# SmartBin — Industrial Zero-Touch Provisioning Architecture

Target: a blank ESP32 powers on, an operator sets WiFi + install details once in
the field, and the device onboards itself to AWS with a unique identity — no
manual certificate copying, scalable to thousands of bins.

Design choices locked for this build:
- **WiFi setup:** ESP-IDF `wifi_provisioning` manager over **SoftAP**, plus a
  **custom endpoint** that also collects the bin's install metadata.
- **Cloud onboarding:** AWS IoT **Fleet Provisioning by Claim**.
- **Identity/metadata:** the **operator enters** bin_id + ward/route/GPS in the
  portal; the device passes them as provisioning parameters.
- **Keys:** **AWS-generated** keypair delivered during claim (CreateKeysAndCertificate).

---

## 1. Two certificates, two phases (the core idea)

A device has **two** identities in its life:

1. **Claim certificate** — a *shared* bootstrap cert baked into the firmware
   image. It can do almost nothing: only talk to the Fleet Provisioning MQTT
   topics. Every device ships with the same claim cert.
2. **Permanent device certificate** — *unique per device*, generated by AWS the
   first time the device provisions itself, then stored in the device's NVS
   flash. All real telemetry uses this.

The claim cert's only job is to let a brand-new device say "hi, make me a real
identity." Once it has a real identity, the claim cert is never used again.

```
Factory image (same for all):  firmware + CLAIM cert (low privilege)
        │  first boot, after WiFi
        ▼
Fleet Provisioning exchange  ──►  AWS issues UNIQUE device cert + key
        │
        ▼
Device stores unique cert in NVS, reconnects as itself, sends telemetry
```

## 2. Device boot state machine

```
        ┌──────────────┐
        │    BOOT      │
        └──────┬───────┘
               ▼
     ┌───────────────────┐   no creds
     │ WiFi creds in NVS? ├──────────────┐
     └─────────┬─────────┘               ▼
               │ yes              ┌─────────────────┐
               ▼                  │ PROVISION_WIFI  │  SoftAP "SmartBin-Setup-XXXX"
     ┌───────────────────┐        │ + captive portal│  operator enters WiFi + bin meta
     │   CONNECT_WIFI    │        └────────┬────────┘
     └─────────┬─────────┘                 │ saved to NVS
               │ connected                 ▼
               ▼                    (reboot / continue)
     ┌───────────────────┐   no device cert
     │ Device cert in NVS?├──────────────┐
     └─────────┬─────────┘               ▼
               │ yes            ┌──────────────────────┐
               ▼                │ FLEET_PROVISION      │ connect w/ CLAIM cert
     ┌───────────────────┐      │  CreateKeys+Register │ pass bin metadata params
     │   MQTT_CONNECT    │      └──────────┬───────────┘
     │  (permanent cert) │                 │ store unique cert in NVS
     └─────────┬─────────┘                 │
               ▼                            └────────► (reconnect as device)
     ┌───────────────────┐
     │    OPERATIONAL    │  read sensors → publish telemetry every 5s
     │  (+ reconnect,    │  handle OTA, deep sleep (optional)
     │   sensor fusion)  │
     └───────────────────┘
```

States, explicitly:
- `BOOT` → init NVS, GPIO, logging.
- `PROVISION_WIFI` → only if no WiFi creds. Start `wifi_provisioning` SoftAP +
  custom `installation` endpoint. Persist WiFi + bin metadata to NVS.
- `CONNECT_WIFI` → STA mode using stored creds; retry/backoff.
- `FLEET_PROVISION` → only if no device cert. Connect with claim cert, run the
  claim exchange, store unique cert/key/thing-name in NVS.
- `MQTT_CONNECT` → connect with the permanent device cert.
- `OPERATIONAL` → sensor fusion + telemetry; auto-reconnect; OTA check.

A factory-reset (long-press a button) clears NVS and returns to `PROVISION_WIFI`.

## 3. WiFi provisioning (field UX)

Using the ESP-IDF `wifi_provisioning` manager with the **SoftAP** scheme:
- Device advertises AP `SmartBin-Setup-<last3 of MAC or bin hint>`.
- Operator connects, uses the Espressif "ESP SoftAP Prov" app **or** the
  provided web page to submit WiFi SSID/password.
- We register a **custom endpoint** (`installation`) via
  `wifi_prov_mgr_endpoint_register()` that accepts a small JSON blob:
  `{ "bin_id", "ward", "route", "latitude", "longitude", "place" }`.
- Both WiFi creds (handled by the manager) and the metadata (our endpoint) are
  written to **NVS** before we tear down the AP.

Security: provisioning is protected with a Proof-of-Possession (PoP) string and
Security Version 2 (SRP6a salted password) so a passer-by can't hijack setup.

## 4. Fleet Provisioning exchange (cloud onboarding)

Provisioning **by claim**, AWS-generated keys, over MQTT reserved topics:

```
Device (claim cert)                         AWS IoT
   │  connect (TLS, claim cert)
   │ ───────────────────────────────────────►
   │  PUBLISH  $aws/certificates/create/json
   │ ───────────────────────────────────────►
   │  SUBSCRIBE .../create/json/accepted
   │ ◄───────────────────────────────────────  certificatePem, privateKey,
   │                                            certificateOwnershipToken
   │  PUBLISH  $aws/provisioning-templates/
   │           SmartBinFleetTemplate/provision/json
   │           { ownershipToken, parameters:{bin_id,ward,route,lat,lng,place} }
   │ ───────────────────────────────────────►
   │                         ┌──────────────────────────────┐
   │                         │ Pre-provisioning hook Lambda  │ validate + dedupe
   │                         │  -> allowProvisioning: true   │ + write registry
   │                         └──────────────────────────────┘
   │  SUBSCRIBE .../provision/json/accepted
   │ ◄───────────────────────────────────────  thingName, deviceConfiguration
   │  store cert+key+thingName in NVS, disconnect, reconnect as device
```

The **provisioning template** maps parameters → a Thing named
`esp32-smartbin-<bin_id>`, attaches the **scoped device policy**, and sets Thing
attributes (ward, route). The **pre-provisioning hook** is the gatekeeper: it
rejects duplicate/invalid bins and writes the registry row, so a device can't
self-register with a bad or duplicate id.

## 5. Security model

| Concern | Mitigation |
|--------|-----------|
| Claim cert is shared / could leak | Claim policy allows ONLY the provisioning topics — it cannot publish telemetry or subscribe to other bins. Rotate/deactivate if leaked. |
| Rogue device self-registering | Pre-provisioning hook validates bin_id format, checks duplicates, can require an allow-list/secret; returns allowProvisioning:false to deny. |
| Per-device blast radius | Scoped device policy restricts each Thing to its OWN topics `smartbin/${iot:Connection.Thing.ThingName}/...` using policy variables. |
| Key delivery | Permanent key is delivered once over TLS during claim; stored in NVS (enable NVS encryption / flash encryption in production). |
| Field setup hijack | wifi_provisioning PoP + Security v2 (SRP6a). |
| Firmware tampering | Enable Secure Boot v2 + Flash Encryption on production boards. |

> Note: today's `SmartBinPolicy` uses `Resource:"*"`. The new **device policy**
> below scopes each device to its own topics with policy variables — adopt it
> for the fleet.

## 6. Registry & data model changes

- `SmartBinRegistry` (existing) stays the source of truth for bin metadata.
  The **pre-provisioning hook** writes the row (idempotent conditional put),
  so registration happens automatically at first onboarding instead of via the
  Admin form. The Admin form remains for manual/back-office edits.
- New optional fields per bin: `provisioned_at`, `fw_version`, `last_seen`,
  `health` (derived). `last_seen`/`health` can be updated by the telemetry
  Lambda.
- Add a `status` lifecycle field: `PROVISIONED → ACTIVE → MAINTENANCE → RETIRED`.

## 7. Device lifecycle management

```
MANUFACTURED ─► (flash factory image w/ claim cert)
   ▼
DEPLOYED ─► operator WiFi+meta setup ─► SELF-PROVISIONED (unique cert)
   ▼
ACTIVE ─► telemetry + OTA updates
   ▼
MAINTENANCE (battery/sensor swap)  ◄──►  ACTIVE
   ▼
RETIRED ─► deactivate device cert, mark registry RETIRED
```

Operations:
- **Decommission:** deactivate+delete the device cert, set registry `RETIRED`.
- **Re-deploy:** factory-reset clears NVS → device re-provisions as a new/renamed bin.
- **Fleet audit:** list Things by the template / by Thing group.

---

## 8. OTA-ready architecture

- Build with the ESP-IDF OTA partition table (`ota_0`/`ota_1` + `otadata`).
- Use **AWS IoT Jobs** to push firmware: a Job document points to a firmware
  binary in S3; device subscribes to its Jobs topics, downloads via HTTPS,
  writes to the inactive OTA slot, validates, and reboots.
- Gate rollouts with Thing Groups + Job rollout/abort config (canary first).
- Keep a "known-good" rollback slot; mark image valid only after a successful
  MQTT connect post-update.

---

## 9. Scaling to thousands

- **Connectivity:** WiFi today; the firmware's transport layer is abstracted so
  LoRa/NB-IoT/LTE can replace the MQTT transport later without touching the
  state machine or sensor logic.
- **Topics:** all per-device, `smartbin/<bin_id>/telemetry|alert|status` — the
  IoT Rule already uses the `+` wildcard, so no per-bin rule changes.
- **Storage:** for high write volume, switch the telemetry table to a
  time-series pattern (partition key `bin_id`, sort key `ts`) or stream to
  Timestream; keep a "latest" item for the dashboard.
- **Throttling:** Fleet Provisioning has account limits; provision in waves and
  pre-create claim certs per region.
- **Dashboards:** paginate the registry; the dashboard should query by route /
  ward rather than scanning all bins.

## 10. Build roadmap (what we implement next)

1. AWS: provisioning template + claim policy + device policy  (`fleet_provisioning_template.json`, `claim_policy.json`, `device_policy.json`).
2. AWS: pre-provisioning hook Lambda (`lambda_preprovision_hook.py`).
3. AWS: deploy script that creates template, claim cert, policies, hook + permission (`deploy_fleet_provisioning.sh` / `.ps1`).
4. Firmware: boot state machine + wifi_provisioning + fleet provisioning + NVS (`main_provisioning.c`).
5. Rollout & operations guide.

Each is delivered as its own file so you can adopt them incrementally without
disturbing the working v2 telemetry pipeline.
